# 2.1 Chart of Accounts - Business Blueprint

## 2.1.1 Business Purpose & Scope

### Business Purpose
The Chart of Accounts serves as the foundational structure for the entire financial management system, providing a systematic framework for classifying, recording, and reporting all financial transactions according to standardized accounting principles.

### Scope
- Comprehensive account structure design and management
- Multi-dimensional account classification and organization
- GAAP/IFRS compliant account categorization
- Hierarchical account relationships and roll-up structures
- Account lifecycle management from creation to archival

### Target Users
- CFO and Finance Directors
- Financial Controllers
- Accounting Managers
- External Auditors
- Regulatory Compliance Teams
- System Administrators

### Business Objectives
- Ensure accurate financial reporting and regulatory compliance
- Provide flexible structure for multi-entity and multi-currency operations
- Support detailed financial analysis and decision-making
- Maintain audit trail and control over account management
- Enable scalability for business growth and complexity

## 2.1.2 Process Flow & Integration

### Account Creation Process
1. **Account Request Submission**
   - Business user submits account creation request
   - Required information: Account type, description, parent account
   - Initial validation for duplicates and compliance

2. **Review and Approval Workflow**
   - Finance Manager review for business justification
   - Controller approval for accounting compliance
   - System Administrator validation for technical requirements

3. **Account Setup and Configuration**
   - Account number assignment based on numbering scheme
   - Default properties and validation rules setup
   - Integration with related modules and reporting structures

4. **Activation and Testing**
   - Account activation in production environment
   - Transaction posting validation
   - Report generation testing

### Account Modification Process
1. **Change Request Initiation**
   - Modification request with business justification
   - Impact analysis on historical and future transactions

2. **Approval and Validation**
   - Multi-level approval based on change significance
   - Compliance validation for regulatory requirements

3. **Implementation and Testing**
   - Controlled implementation with rollback capability
   - Transaction posting validation
   - Historical data impact assessment

### Account Archival Process
1. **Eligibility Assessment**
   - Zero balance verification
   - Historical transaction review
   - Regulatory compliance check

2. **Archival Execution**
   - Account status change to inactive
   - Historical data preservation
   - Reporting structure updates

### Integration Points
- **General Ledger**: Real-time account validation and posting
- **Accounts Payable/Receivable**: Automatic account assignment
- **Fixed Assets**: Asset account integration and depreciation
- **Cash Management**: Bank and cash account reconciliation
- **Budgeting**: Budget account mapping and tracking
- **Reporting**: Financial statement account mapping
- **Tax Management**: Tax account classification and reporting

## 2.1.3 Data Model & Schema

### Core Account Master Table Structure

**GL_ACCOUNTS**
- **account_id** (Primary Key, UUID)
- **account_number** (VARCHAR(20), Unique, Indexed)
- **account_name** (VARCHAR(200), Required)
- **account_description** (TEXT)
- **account_type** (ENUM: 'ASSET', 'LIABILITY', 'EQUITY', 'REVENUE', 'EXPENSE')
- **account_category** (VARCHAR(50))
- **account_subcategory** (VARCHAR(50))
- **parent_account_id** (Foreign Key to GL_ACCOUNTS)
- **account_level** (INTEGER)
- **normal_balance** (ENUM: 'DEBIT', 'CREDIT')
- **currency_code** (VARCHAR(3), Default 'USD')
- **status** (ENUM: 'ACTIVE', 'INACTIVE', 'PENDING', 'CLOSED')
- **creation_date** (DATETIME)
- **effective_date** (DATE)
- **closure_date** (DATE)
- **created_by** (Foreign Key to USER_TABLE)
- **approved_by** (Foreign Key to USER_TABLE)
- **approval_date** (DATETIME)
- **last_modified_date** (DATETIME)
- **modified_by** (Foreign Key to USER_TABLE)

### Account Hierarchy Table

**GL_ACCOUNT_HIERARCHY**
- **hierarchy_id** (Primary Key, UUID)
- **parent_account_id** (Foreign Key to GL_ACCOUNTS)
- **child_account_id** (Foreign Key to GL_ACCOUNTS)
- **hierarchy_level** (INTEGER)
- **rollup_percentage** (DECIMAL(5,2), Default 100.00)
- **effective_start_date** (DATE)
- **effective_end_date** (DATE)
- **created_date** (DATETIME)

### Account Properties Table

**GL_ACCOUNT_PROPERTIES**
- **property_id** (Primary Key, UUID)
- **account_id** (Foreign Key to GL_ACCOUNTS)
- **property_name** (VARCHAR(100))
- **property_value** (VARCHAR(500))
- **property_type** (ENUM: 'TEXT', 'NUMBER', 'DATE', 'BOOLEAN')
- **is_required** (BOOLEAN, Default FALSE)
- **validation_rule** (TEXT)
- **created_date** (DATETIME)
- **modified_date** (DATETIME)

### Account Validation Rules Table

**GL_ACCOUNT_VALIDATION**
- **validation_id** (Primary Key, UUID)
- **account_id** (Foreign Key to GL_ACCOUNTS)
- **validation_type** (ENUM: 'RANGE', 'LIST', 'FORMULA', 'CUSTOM')
- **validation_rule** (TEXT)
- **error_message** (VARCHAR(500))
- **is_active** (BOOLEAN, Default TRUE)
- **created_date** (DATETIME)

### Account History Table

**GL_ACCOUNT_HISTORY**
- **history_id** (Primary Key, UUID)
- **account_id** (Foreign Key to GL_ACCOUNTS)
- **action_type** (ENUM: 'CREATE', 'MODIFY', 'ACTIVATE', 'DEACTIVATE', 'CLOSE')
- **old_values** (JSON)
- **new_values** (JSON)
- **change_reason** (TEXT)
- **changed_by** (Foreign Key to USER_TABLE)
- **change_date** (DATETIME)

### Data Relationships and Constraints

**Primary Constraints:**
- Unique constraint on account_number within entity
- Foreign key relationships maintain referential integrity
- Check constraints for account_type and normal_balance consistency

**Indexing Strategy:**
- Primary index on account_id
- Unique index on account_number
- Composite index on account_type and status
- Index on parent_account_id for hierarchy queries

**Data Validation:**
- Account number format validation (alphanumeric with specific patterns)
- Account type and normal balance consistency checks
- Hierarchy level validation to prevent circular references
- Effective date validation for temporal integrity
